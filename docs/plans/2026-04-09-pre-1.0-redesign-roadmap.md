# Pre-1.0 Redesign Roadmap

## Goal

Design and implement the library correctly before `1.0.0`.

This roadmap assumes:

- correctness matters more than API continuity
- internal clarity matters more than incremental migration
- public APIs may be redesigned freely
- partial compatibility layers are not required unless they simplify implementation

The desired end state is a clean, defensible architecture for:

- adapter initialization and capability detection
- protocol probing and selection
- vehicle discovery and profiling
- module visibility and physical routing
- robust communication across older and newer vehicles
- robust communication across USB and BLE adapters

## Architectural Principles

1. `Session` is the primary public API surface.
2. Discovery is a first-class lifecycle, not an incidental step.
3. Logical module names are resolved in `Session`, not in adapters.
4. Adapters accept resolved physical routing only.
5. Transports move bytes only; they do not own protocol logic.
6. Protocol framing/decoding should be isolated from adapter setup logic.
7. Single-flight request execution is mandatory for real adapters.
8. Debuggability is a core feature, not an afterthought.

## Target Architecture

### Public Surface

- `Session`
  - initialize and discover adapter/vehicle
  - query connection state
  - query discovery profile
  - perform standard OBD operations
  - perform enhanced/module-targeted operations
  - perform diagnostic session operations
  - expose visible ECUs and routing information

### Internal Boundaries

- `transport`
  - serial
  - BLE
  - logging/capture

- `adapter`
  - adapter initialization
  - physical command emission
  - routed request execution
  - adapter-specific quirks and sanitization

- `session`
  - lifecycle
  - discovery profile
  - routing resolution
  - stateful orchestration

- `protocol`
  - bus/protocol framing
  - multi-frame assembly rules
  - ECU/source extraction
  - payload decoding helpers

- `vehicle/specs`
  - identity matching
  - communication topology
  - module addressing
  - enhanced PID definitions
  - rules and thresholds

## Non-Goals

- preserving pre-1.0 public API shape
- retaining adapter-first helper APIs if they conflict with the target architecture
- deferring correctness work because of hypothetical downstream breakage

## Major Redesign Areas

### Area 1: Session-First API

Outcome:

- all meaningful user-facing operations live on `Session`
- standalone adapter-first helper flows are removed or reduced to internal utilities

Checklist:

- [x] Make `Session` the documented primary API
- [x] Add all diagnostic session operations to `Session`
- [x] Add all module-targeted enhanced operations to `Session`
- [x] Remove public reliance on raw adapter access for normal usage
- [x] Remove or internalize adapter-first helper flows that bypass discovery
- [x] Update docs and examples to use only the session-first surface

### Area 2: Discovery Lifecycle

Outcome:

- initialization becomes a real lifecycle with explicit states and recorded outcomes

Checklist:

- [x] Define a connection/discovery state enum
- [x] Add state to `Session`
- [ ] Distinguish:
  - adapter present
  - adapter initialized
  - ignition off
  - protocol negotiating
  - connected
  - unsupported protocol
  - disconnected/error
- [x] Add `Session::connection_state()`
- [x] Record state transitions during initialization and discovery
- [x] Test all major lifecycle transitions

### Area 3: Discovery Profile

Outcome:

- a single profile describes what was discovered and how the library decided to talk to the vehicle

Checklist:

- [x] Keep `DiscoveryProfile` as authoritative session discovery state
- [x] Include adapter capabilities
- [x] Include selected protocol
- [x] Include active bus
- [x] Include resolved modules and addresses
- [x] Include protocol probe outcomes
- [x] Include visible ECUs observed during communication
- [x] Include source of protocol choice:
  - auto-detect
  - spec preference
  - fallback probe
- [x] Expose `Session::discovery()`

### Area 4: Routed Request Model

Outcome:

- adapters only receive resolved physical routing, never logical module names

Checklist:

- [x] Remove logical module routing from the adapter execution path
- [x] Use a single internal routed request type
- [x] Make session-side request resolution mandatory for module-targeted calls
- [x] Resolve module names from `DiscoveryProfile`
- [x] Fail explicitly when:
  - no discovery profile exists
  - module is unknown
  - module has no address
  - module bus conflicts with active bus
- [x] Ensure broadcast requests remain simple and explicit

- [x] Define CAN configuration policy for:

- [x] Decide which CAN options are core defaults, spec-driven, or explicitly unsupported

## Phased Execution

Goal:

- define the correct pre-1.0 API surface before more implementation spreads across the old model

Checklist:

- [x] Define the session-first public API surface
- [x] Define the final routed-request internal boundary
- [x] Decide which current helper modules become internal-only
- [x] Update architecture docs to reflect the target shape

Done when:

- there is one authoritative architecture to implement against

## Phase 2: Finish Routing And Diagnostic Session Ownership

Goal:

- complete session-owned module-targeted routing

Checklist:

- [x] Finish unresolved-routing error model
- [x] Complete targeted routing tests including switching modules
- [x] Move all diagnostic session flows to `Session`
- [x] Remove remaining direct `adapter.request(...)` module-target paths
- [x] Confirm broadcast behavior remains correct after targeted operations

Done when:

- no normal library flow depends on unresolved module targets reaching an adapter

## Phase 3: Add Lifecycle And Probe Policy

Goal:

- make discovery stateful and robust for older vehicles

Checklist:

- [x] Record probe decisions in discovery profile

- [x] Test older-vehicle fallback scenarios

- initialization and protocol selection are explicit, observable, and testable

- stop regressions across vehicle families and adapter types before `1.0.0`

## Success Criteria

This roadmap is complete when:

- the library’s public API is coherent and session-first
- the internals match the public API instead of working around it
- older and newer vehicles can be discovered through an explicit policy
- USB and BLE adapter differences are contained below the session layer
- module-targeted operations are physically routed correctly
- adapter quirks and failures are diagnosable from tests and logs
- the project is technically ready for a defensible `1.0.0`
